# 人物誌建立指南

> 供 ux-empathize skill 的 Step 1（Identify Users）參考。

## 從研究到 Persona 的合成流程

```
研究資料收集 → 行為變數辨識 → 行為模式聚類 → Persona 建立 → 設計應用
```

### 1. 行為變數辨識

從研究中提取影響設計的**行為差異**，而非人口統計差異：

| 行為變數類型 | 範例 |
|---|---|
| 使用頻率 | 每天用 vs 偶爾用 |
| 技術熟悉度 | 熟練快捷鍵 vs 只用基本功能 |
| 任務目標 | 快速完成 vs 仔細確認 |
| 決策模式 | 獨立判斷 vs 需要主管核准 |
| 資訊需求 | 只看摘要 vs 需要完整細節 |

### 2. 行為模式聚類

將有相似行為變數組合的使用者歸為同一群：
- 同一群內的使用者：行為模式相似，可用同一個 Persona 代表
- 不同群之間：至少有 2-3 個行為變數明顯不同

### 3. Persona 模板

每個 Persona 應包含：

```markdown
## {Persona 名稱}

**角色**：{職位/身份}
**使用頻率**：{每天/每週/偶爾}
**技術熟悉度**：{高/中/低}

### 目標
- {主要目標}：{具體描述}
- {次要目標}：{具體描述}

### 行為模式
- {典型的操作方式}
- {偏好的互動模式}
- {資訊消費習慣}

### 挫折點
- {最大的痛點}
- {常見的卡關情境}

### 使用情境
- {何時使用}
- {在什麼環境下使用}
- {使用什麼裝置}

### 設計影響
- {這個 persona 的需求如何影響功能優先級}
- {需要特別注意的設計考量}
```

## Persona 類型

| 類型 | 說明 | 設計權重 |
|------|------|---------|
| **Primary** | 核心使用者，設計的首要對象 | 最高 — 所有設計決策優先考慮 |
| **Secondary** | 偶爾使用，需求不能與 Primary 衝突 | 中 — 在不影響 Primary 的前提下滿足 |
| **Edge-case** | 極端使用情境 | 低 — 記錄但不主導設計 |

## Anti-Patterns

| Anti-pattern | 問題 | 替代做法 |
|---|---|---|
| 純人口統計 Persona | 「25 歲男性」不影響設計決策 | 聚焦行為差異（使用頻率、目標、技術程度） |
| 虛構敘事式 | 「小明喜歡在週末逛網站」沒有研究依據 | 每個特徵需有觀察或訪談佐證 |
| 太多 Persona | >3 個導致設計失焦 | Primary 1-2 個 + Secondary 1 個 |
| 淺層差異化 | 兩個 Persona 行為幾乎一樣 | 合併，或找出真正影響設計的差異 |
| 建完就丟 | Persona 沒有連結到設計決策 | 每個設計選擇標註「這服務哪個 Persona 的哪個需求」 |

## Persona → 設計決策的連結

Persona 的終極目標是**影響設計決策**：

| Persona 特徵 | 設計影響 |
|---|---|
| 高頻使用者 | 提供快捷操作、記住上次設定 |
| 低技術熟悉度 | 明確的引導文字、避免專業術語 |
| 時間壓力大 | 減少步驟、預設合理值、批次操作 |
| 需要主管核准 | 審核流程、狀態追蹤、通知機制 |
| 多裝置切換 | 資料同步、響應式設計 |
