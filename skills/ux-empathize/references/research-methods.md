# 使用者研究方法論

> 供 ux-empathize skill 的 Step 2（Discover Pain Points）參考。

## 研究方法選擇

| 方法 | 適用場景 | 產出 |
|------|---------|------|
| **使用者訪談** | 了解動機、行為、痛點 | 洞察 + 引述 |
| **情境觀察** | 了解真實使用環境 | 行為模式 + 環境因素 |
| **任務分析** | 拆解使用者完成任務的步驟 | 步驟流程 + 摩擦點 |
| **競品分析** | 了解市場現有解法 | 機會點 + 差異化方向 |

## 訪談執行指南

### 準備階段
1. **定義研究問題**：這次訪談想釐清什麼？（不超過 3 個核心問題）
2. **準備訪談大綱**：5-8 個開放式問題，從廣泛到具體
3. **招募對象**：每個角色 3-5 人足以發現 80% 的痛點（Nielsen, 2000）

### 提問原則

**好的問題**（開放式、行為導向）：
- 「描述一下你上次完成這個任務的過程？」
- 「遇到什麼困難？你怎麼解決的？」
- 「哪個步驟花最多時間？」
- 「如果可以改一件事，你會改什麼？」

**避免的問題**（引導式、封閉式）：
- ❌「你覺得這個功能好不好用？」（引導 + 封閉）
- ❌「如果我們加一個篩選功能，你會用嗎？」（引導解法）
- ❌「這個流程是不是太複雜了？」（暗示答案）

### 萃取洞察

1. **記錄行為，不只記錄意見**：使用者做了什麼 > 使用者說了什麼
2. **標記情緒信號**：猶豫、嘆氣、反覆操作 = 潛在痛點
3. **區分表面抱怨 vs 根本原因**：「搜尋太慢」可能是「不知道怎麼用篩選條件」
4. **交叉驗證**：同一個痛點被 ≥2 人提到 → 高可信度

## 5W1H 研究框架

| 維度 | 問什麼 | 設計意義 |
|------|--------|---------|
| **Who** | 誰在用？（角色、頻率、技術程度） | Persona 定義 |
| **What** | 在做什麼任務？ | 功能範圍 |
| **When** | 什麼時候用？（時段、頻率） | 使用情境 |
| **Where** | 在哪裡用？（環境、裝置） | 平台 + 環境限制 |
| **Why** | 為什麼這樣做？（動機） | 需求優先級 |
| **How** | 怎麼完成的？（步驟） | 流程優化機會 |

## Anti-Patterns

| Anti-pattern | 問題 | 替代做法 |
|---|---|---|
| 問卷式訪談 | 只得到表面數據，沒有深度洞察 | 用開放式追問挖掘「為什麼」 |
| 引導性問題 | 得到你想聽的答案，不是真相 | 用中性語言、讓使用者自己描述 |
| 只問主管 | 主管觀點 ≠ 實際使用者體驗 | 直接訪談一線操作人員 |
| 一次問太多 | 使用者疲勞，回答品質下降 | 每次 30-45 分鐘，聚焦 3 個核心問題 |
| 忽略非語言信號 | 錯過重要的行為線索 | 記錄猶豫、反覆操作、求助行為 |
